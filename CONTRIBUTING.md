# ✍️ 팀 프로젝트 기여 가이드 (CONTRIBUTING)

이 문서는 팀 프로젝트의 코드 안정성과 효율적인 협업을 위한 규칙과 가이드라인을 정의합니다.

**Git/GitHub의 핵심은 'main 브랜치'를 보호하고, 안전한 브랜치에서 작업 후 검토(PR)를 거쳐 합치는 것입니다.**

---

## 1. ⚙️ 기본 작업 흐름 (GitHub Flow)

모든 새로운 작업은 다음 5단계의 흐름을 따릅니다.

| 단계 | 기능/용어 | 목표 | Git 명령어 (참고) |
| :--- | :--- | :--- | :--- |
| **1. 시작** | `git pull` & **Branch** 생성 | 최신 코드를 확보하고, 충돌 없는 나만의 작업 공간을 만듭니다. | `git checkout -b feature/name` |
| **2. 작업** | **Commit** & **Push** | 맡은 기능을 구현하고, 내용을 자주 저장하여 백업합니다. | `git commit -m "feat:..."` |
| **3. 검토 요청** | **Pull Request (PR)** | 작업이 완료되면, 팀원들에게 최종 검토를 요청합니다. | (GitHub 웹에서 생성) |
| **4. 검토** | **Code Review** | 팀원의 피드백을 반영하고 PR에 대한 승인(Approve)을 받습니다. | (GitHub 웹에서 진행) |
| **5. 합치기** | **Merge** | 승인이 완료되면 `main` 브랜치에 코드를 반영합니다. | (GitHub 웹 Merge 버튼) |

---

## 2. 🌳 브랜치 (Branch) 생성 규칙

**반드시 `main` 브랜치에서 분기하여 새로운 브랜치를 생성**해야 하며, 브랜치 이름은 작업 종류를 나타내는 **접두사**와 작업 내용을 포함해야 합니다.

* **포맷:** `접두사/간결한-작업-내용` (띄어쓰기 대신 `-` 사용)
* **예시:** `feature/user-login`, `fix/nav-bar-bug`

| 접두사 | 사용 목적 | 예시 |
| :--- | :--- | :--- |
| **`feature/`** | 새로운 기능 추가 (가장 많이 사용) | `feature/product-detail-page` |
| **`fix/`** | 버그 수정 | `fix/scroll-not-working` |
| **`refactor/`** | 기능 변경 없이 코드 구조 개선/정리 | `refactor/clean-unused-code` |

---

## 3. 💾 커밋 (Commit) 메시지 규칙

커밋 메시지는 코드의 **역사책**과 같습니다. 어떤 변경을 했는지 쉽게 알 수 있도록 일관된 규칙을 따릅니다.

### 1) 포맷

`[타입]: [제목 (무엇을 했는지)]`

* **제목:** 50자 이내의 간결한 문장 (현재형/명령형 사용 권장)

### 2) 타입 (Type) 목록

| 타입 | 의미 | 예시 |
| :--- | :--- | :--- |
| **`feat`** | 새로운 기능 추가 (Feature) | `feat: 사용자 로그인 페이지 UI 구현` |
| **`fix`** | 버그 수정 (Fix) | `fix: 상품 가격 표기 오류 수정` |
| **`docs`** | 문서 수정 (README, 주석 등) | `docs: CONTRIBUTING.md 업데이트` |
| **`style`** | 코드 스타일 변경 (CSS, 포맷 등) | `style: 버튼에 그림자 속성 추가` |
| **`refactor`** | 코드 리팩토링 (Refactoring) | `refactor: 함수 이름 명확하게 변경` |

---

## 4. 💌 Pull Request (PR) 가이드라인

PR은 코드 병합 전 팀원들과 소통하고 검증하는 중요한 단계입니다.

### 1) PR 생성 전 필수 확인 사항

1.  **브랜치 최신화:** PR을 생성하기 전에 반드시 **`main` 브랜치의 최신 내용**을 내 브랜치에 반영했는지 확인합니다.
    * `git pull origin main` 후 충돌이 있다면 해결합니다.
2.  **셀프 테스트:** 내가 수정한 기능이 **의도한 대로 정상 작동**하는지 확인합니다.

### 2) PR 본문 템플릿 (스크린샷 필수)

아래 템플릿을 복사하여 사용하며, 특히 **스크린샷 첨부**는 초심자 리뷰에서 필수입니다.

```markdown
### 🚀 변경 사항 요약 (Summary)
[여기에 이번 PR에서 구현하거나 수정한 내용 3줄 이내로 간결하게 작성]

### 💡 변경 상세 내용 (Details)
- [어떤 파일에서 어떤 부분을 수정했는지 구체적으로 작성]
- [새로 추가된 라이브러리/기술이 있다면 명시]

### ✅ 테스트 방법 (How to Test)
1. [ex: npm run start 로 앱 실행]
2. [ex: 로그인 페이지 접속]
3. [ex: ID: test, PW: 1234 입력 후 로그인 버튼 클릭 시 작동 확인]

### 📸 스크린샷 또는 GIF (필수)
[수정된 화면의 실행 전/후를 캡처하여 첨부합니다.]

## 5. 👶 초심자 리뷰 및 검토 방법 (바이브 코딩 최적화)

우리는 복잡한 코드 자체보다 **실행 결과**를 통해 코드를 검토하는 것을 우선합니다.

### 1) 리뷰의 기본 원칙: 코드 < **실행 결과**

* 우리는 복잡한 코딩 스타일이나 알고리즘을 보지 않습니다. **기능이 의도대로 잘 작동하는지**가 최우선입니다.

### 2) 리뷰어 (검토자) 행동 지침

| 행동 | 설명 |
| :--- | :--- |
| **웹사이트 확인** | PR 페이지에서 **스크린샷/GIF**를 먼저 확인하고, 시각적으로 문제가 없는지 판단합니다. |
| **로컬 테스트** | PR 본문의 **`테스트 방법`** 가이드를 따라 자신의 컴퓨터에 코드를 가져와 **실제로 실행해봅니다.** (UI/UX, 기능 오류 중점 확인) |
| **체크리스트 활용** | PR 본문에 있는 **체크리스트(오타, 콘솔 로그 등)**가 모두 체크되었는지 확인합니다. |
| **피드백 방법** | 문제 발견 시, PR에 **"어디서/어떤 문제가 발생했다"**라고 댓글을 남기고, **절대로 직접 코드를 수정하지 않습니다.** |

### 3) 리뷰어 체크리스트 (승인 전 확인)

리뷰어는 승인(Approve) 버튼을 누르기 전에 아래 항목을 확인해야 합니다.

* [ ] 1. PR 본문에 **스크린샷/테스트 가이드**가 모두 작성되었는가?
* [ ] 2. `테스트 방법`대로 실행했을 때 **버그나 화면 깨짐**이 발생하지 않는가?
* [ ] 3. 코드 작성자가 **오타나 불필요한 테스트 코드(console.log 등)**를 삭제했는가?

### 4) 병합 (Merge)

* **리뷰어 1명 이상**의 **`Approve`**를 받은 후에만 PR을 `main` 브랜치에 병합(Merge)할 수 있습니다.
