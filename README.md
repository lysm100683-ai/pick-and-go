## 🤝 팀 협업 워크플로우 (Team Workflow)

팀원 모두가 이 규칙을 따르며 코드를 공유하고 통합합니다.

### 1\. 🥇 기본 원칙: `main` 브랜치 보호

  * **`main`** 브랜치는 **항상 배포 가능한 안정적인 최종 코드**를 유지합니다.
  * **절대로** `main` 브랜치에 직접 커밋(Commit)하거나 푸시(Push)하지 않습니다.

### 2\. 🌳 브랜치 사용 (나만의 작업 공간)

| 구분 | 규칙 | 설명 |
| :--- | :--- | :--- |
| **작업 시작** | **`main`에서 시작** | 작업을 시작할 때마다 `main` 브랜치에서 **최신 상태**를 받고 새로운 **작업 브랜치**를 생성합니다. |
| **이름 규칙** | **`[종류]/[작업 내용]`** | 브랜치 이름은 작업 종류와 내용을 간결하게 표현합니다. <br> 예시: `feat/login-page`, `fix/map-rendering`, `docs/update-readme` |
| **작업 완료** | **삭제 필수** | Pull Request(PR)가 `main`에 병합(Merge)된 후에는 해당 작업 브랜치를 바로 삭제합니다. |

-----

## 3\. 💾 커밋 및 푸시 규칙

| 구분 | 규칙 | 설명 |
| :--- | :--- | :--- |
| **커밋 주기** | **작고 자주** 커밋합니다. | 기능 하나를 완성할 때마다 또는 1시간 이내의 작은 단위 작업이 끝날 때마다 커밋합니다. |
| **메시지 형식** | **`[종류]: [내용 요약]`** | 무슨 작업을 했는지 명확히 알 수 있도록 합니다. <br> 예시: `feat: 사용자 로그인 기능 추가` |
| **메시지 종류** | **필수 4가지** | **`feat`** (새 기능 추가), **`fix`** (버그 수정), **`refactor`** (코드 구조 개선), **`docs`** (문서 수정) |
| **푸시 주기** | **매일 백업** | 최소 하루에 한 번 이상은 `git push`로 작업 내용을 GitHub에 백업합니다. |

-----

## 4\. 💌 Pull Request (PR) 및 리뷰

| 구분 | 규칙 | 설명 |
| :--- | :--- | :--- |
| **PR 생성 전** | **최신화 필수** | PR을 만들기 전에 **반드시** 로컬 **`main` 브랜치**의 최신 내용을 내 작업 브랜치에 `git merge main` 하여 **충돌을 미리 해결**해야 합니다. |
| **PR 내용** | **명확하게 작성** | PR 제목에 **[Feat]** 또는 **[Fix]** 등 종류를 명시하고, 내용에는 \*\*'무엇을 바꿨는지'\*\*와 \*\*'어떻게 테스트했는지'\*\*를 자세히 적습니다. |
| **리뷰 의무** | **최소 1명 승인** | `main` 브랜치에 병합하기 전에 **최소 1명 이상의 팀원**에게 코드 리뷰를 요청하고 \*\*'승인(Approve)'\*\*을 받아야 병합할 수 있습니다. |
| **충돌 해결** | **PR 작성자가 책임** | PR에서 \*\*충돌(Conflict)\*\*이 발생하면, PR을 보낸 **작성자가 직접** 해결하고 다시 푸시(Push)해야 합니다. |
