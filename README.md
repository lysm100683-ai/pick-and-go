# 🚀 [Pick-and-Go] 개발 팀 협업 가이드

이 문서는 우리 팀이 **GitHub**를 활용하여 코드를 공유하고 수정할 때 지켜야 할 **가장 효율적인 룰**을 담고 있습니다.

## 1\. 🥇 핵심 원칙: 브랜치 기반 워크플로우

팀 코드를 \*\*'공유 레시피북'\*\*이라고 생각하고, 이 3가지 원칙을 반드시 지킵니다.

1.  **`main` 브랜치 보호:** **`main`** 브랜치에는 **절대로** 직접 코드를 수정하거나 업로드(Push)하지 않습니다. `main`은 항상 **완벽하게 작동하는 최종 코드**입니다.
2.  **개인 작업 공간 사용:** 모든 기능 개발 및 버그 수정은 **개인 작업 브랜치**를 만들어서 진행합니다.
3.  **검토 후 병합:** 작업이 완료되면 반드시 \*\*Pull Request (PR)\*\*를 통해 팀원의 검토를 거친 후 `main`에 합칩니다.

-----

## 2\. 🌳 브랜치 사용 및 관리 규칙

| 구분 | 규칙 | 설명 |
| :--- | :--- | :--- |
| **작업 시작** | **`main`에서 최신화** | 작업을 시작할 때마다 `main` 브랜치에서 최신 코드를 당겨와 (Pull) 새로운 작업 브랜치를 생성합니다. |
| **이름 규칙** | **`[종류]/[작업 내용]`** | 브랜치 이름은 작업 **종류**와 **내용**을 간결하게 표현합니다. <br> 예시: `feat/login-page-ui`, `fix/map-rendering-bug` |
| **작업 완료** | **브랜치 삭제** | PR이 `main`에 병합(Merge)된 후에는 해당 작업 브랜치를 **바로 삭제**하여 저장소를 깨끗하게 유지합니다. |

-----

## 3\. 💾 코드 저장 및 메모 규칙 (Commit & Push)

개인의 수정 내용을 \*\*'메모'\*\*하는 장소는 **커밋 메시지**입니다.

| 구분 | 규칙 | 설명 | 작성 장소 |
| :--- | :--- | :--- | :--- |
| **저장 주기** | **작고 자주** 커밋합니다. | 기능 하나를 완성할 때마다 또는 1시간 이내의 작은 단위 작업이 끝날 때마다 저장합니다. | **Commit Message** |
| **메시지 형식** | **`[종류]: [내용 요약]`** | **무엇을 했는지** 명확히 알 수 있도록 기록합니다. <br> **종류 예시:** `feat` (새 기능), `fix` (버그 수정), `refactor` (코드 개선) | **Commit Message** |
| **푸시 주기** | **매일 백업** | 최소 하루에 한 번 이상은 `git push`로 작업 내용을 GitHub에 업로드하여 백업합니다. | **Terminal/IDE** |

-----

## 4\. 💌 팀 공유 및 검토 규칙 (Pull Request)

팀원들에게 \*\*'검토 요청'\*\*하는 설명은 **Pull Request (PR) 본문**에 작성합니다.

| 구분 | 규칙 | 설명 | 작성 장소 |
| :--- | :--- | :--- | :--- |
| **PR 생성 전** | **최신화 및 충돌 해결** | PR을 만들기 전에 **반드시** 로컬 `main`의 최신 내용을 내 브랜치에 합쳐서 **충돌(Conflict)을 미리 해결**합니다. | **Terminal/IDE** |
| **PR 내용** | **명확하게 작성** | **1. 변경 사항 요약, 2. 테스트 방법**을 자세히 적어 팀원들이 쉽게 확인할 수 있도록 합니다. (UI 변경 시 스크린샷 필수) | **PR 본문** |
| **리뷰 의무** | **최소 1명 승인** | `main` 브랜치에 병합하려면 **최소 1명 이상의 팀원**에게 \*\*'승인(Approve)'\*\*을 받아야 합니다. | **GitHub PR 페이지** |
| **충돌 발생 시** | **작성자 책임** | PR에서 충돌이 발생하면, PR을 보낸 **작성자**가 직접 해결하고 다시 푸시해야 합니다. | **Terminal/IDE** |
